Extending the Platform

Step-by-step guides for adding a new data source, KPI, domain configuration, agent, or insight rule — without touching platform core code. Every extension point is an interface or a configuration object. New capabilities are additions, never modifications.

In This Guide
1. Add a New Data Source

Implement DataConnector for any source (DB2, Oracle, Snowflake, REST). Register it. No platform code changes.

2. Add a New KPI

Declare a KpiDefinition in the schema registry. The KPI Engine executes it automatically. Zero new endpoints.

3. Add a New Domain

Create a DomainConfig object. Map any business domain (HR, Finance, Supply Chain) to the platform's abstract schema.

4. Add a New Insight Rule

Register an InsightRuleDef. The deterministic InsightEngine picks it up automatically. No changes to InsightEngine itself.

5. Add a New Agent

Implement the Agent interface. Register it with the Orchestrator. Emits standard AgentResult + RunTrace.

6. Switch the LLM

Implement NarratorService for any LLM (GPT-4, Llama, Granite, Claude). Swap the binding. Zero changes to calling code.

01 · Add a New Data Source

Implementing a DataConnector

Any new data source requires exactly one new class implementing the DataConnector interface, and one registration call. The entire platform above Layer 1 is unaffected.

Contract
Your connector must implement connect(), fetch(), status(), and disconnect(). The fetch() method must return a valid DataSet with a populated schema, rows, checksum, and fetchedAt timestamp. All column types must be coerced at ingest — never at query time.
Create the connector class

Extend the abstract BaseConnector to inherit boilerplate (logging, retry logic, status tracking).

// src/connectors/SnowflakeConnector.ts
import { BaseConnector, DataSet, ConnectorQuery, ColumnSchema } from "../platform/connectors";
import { createConnection } from "snowflake-sdk";

export class SnowflakeConnector extends BaseConnector {
  readonly id = "snowflake";
  readonly displayName = "Snowflake Data Warehouse";

  private conn: snowflake.Connection | null = null;

  constructor(private config: SnowflakeConfig) {
    super();
  }

  async connect(): Promise<void> {
    this.conn = createConnection({
      account:   this.config.account,
      username:  this.config.username,
      privateKey: this.config.privateKeyPath,  // key-pair auth (no password in env)
      database:  this.config.database,
      warehouse: this.config.warehouse,
    });
    await this.conn.connect();
    this.markHealthy();
  }

  async fetch(query: ConnectorQuery, schema: ColumnSchema[]): Promise<DataSet> {
    if (!this.conn) throw new ConnectorError("Not connected. Call connect() first.");

    // Build SQL from the ConnectorQuery (table name + optional WHERE clause)
    const sql = query.rawSql ?? this.buildSql(query, schema);

    const rows = await this.executeQuery(sql);

    // BaseConnector.normalizeRows() coerces types, validates against schema,
    // collects warnings, and computes the SHA-256 checksum
    return this.normalizeRows(rows, schema, query);
  }

  async status() {
    return this.conn
      ? { healthy: true, lastFetchedAt: this.lastFetchedAt, rowCount: this.lastRowCount }
      : { healthy: false, error: "Not connected" };
  }

  async disconnect(): Promise<void> {
    await this.conn?.destroy();
    this.conn = null;
  }

  private buildSql(query: ConnectorQuery, schema: ColumnSchema[]): string {
    const cols = schema.map(c => c.aliases?.[0] ?? c.name).join(", ");
    const where = query.filter ? `WHERE ${query.filter}` : "";
    return `SELECT ${cols} FROM ${query.table} ${where}`;
  }
}
Register the connector
// src/platform/ConnectorRegistry.ts
import { SnowflakeConnector } from "../connectors/SnowflakeConnector";

ConnectorRegistry.register("snowflake", (config) => new SnowflakeConnector(config));
// That's it. The platform will resolve "snowflake" from any DomainConfig that references it.
Reference in a DomainConfig
// Any domain can now use the Snowflake connector as its data source
const SUPPLY_CHAIN_DOMAIN: DomainConfig = {
  id: "supply_chain",
  dataSources: [
    {
      connectorId: "snowflake",
      role: "inventory",
      config: {
        account: process.env.SNOWFLAKE_ACCOUNT,
        username: process.env.SNOWFLAKE_USER,
        database: "SUPPLY_CHAIN_DB",
        warehouse: "ANALYTICS_WH",
      },
      query: { table: "INVENTORY_FACTS", filter: "YEAR = CURRENT_YEAR" },
      refreshIntervalMs: 3_600_000,  // hourly
    }
  ]
};
Write the connector test

Every connector must pass the standard ConnectorContractTest suite, which validates that fetch() returns a well-formed DataSet with the correct schema, checksum, and type coercions. 

// src/connectors/__tests__/SnowflakeConnector.test.ts
import { ConnectorContractTest } from "../../platform/testing";

// Runs 12 standard contract assertions automatically
ConnectorContractTest.run(
  () => new SnowflakeConnector(TEST_CONFIG),
  SNOWFLAKE_FIXTURE_QUERY,
  EXPECTED_SCHEMA
);
No-touch guarantee
After these 4 steps, the Snowflake connector is available to every domain config and every layer above Layer 1. The KPI Engine, Intelligence Layer, Narrator, and all agents are completely unaware that a new source type was added.
02 · Add a New KPI

Declaring a KpiDefinition Zero Code Change

KPIs are declared, not coded. The AggregationEngine reads the SchemaRegistry and executes every registered KpiDefinition automatically. There is no "add a new endpoint" step — the REST API exposes all registered KPIs dynamically.

Declare the KpiDefinition
// src/kpi/definitions/backlog.kpi.ts
import { KpiDefinition } from "../../platform/kpi";

export const BACKLOG_KPI: KpiDefinition = {
  id: "backlog",
  displayName: "Order Backlog",
  unit: "currency_usd",
  aggregation: "sum",
  sourceColumn: "Backlog Value",          // must match a column in the DataSet schema
  filterDimensions: ["geo", "market", "quarter", "year", "offeringArea"],
  comparisons: [
    { type: "QoQ", priorPeriodFn: "prevQuarter" },
    { type: "YoY", priorPeriodFn: "sameQtrPriorYear" },
  ],
  thresholds: [
    { condition: "qoq_delta < -15", severity: "HIGH",  label: "Backlog declining rapidly" },
    { condition: "qoq_delta < -5",  severity: "MED",   label: "Backlog declining" },
    { condition: "value === 0",      severity: "HIGH",  label: "No backlog recorded" },
  ],
  insightRules: [
    { id: "backlog_geo_concentration", description: "Flag if 1 geo holds >60% of total backlog" },
    { id: "backlog_pipeline_coverage", description: "Compare backlog to active pipeline as coverage ratio" },
  ],
};
Register in the SchemaRegistry
// src/kpi/SchemaRegistry.ts  (or domain-level config)
import { BACKLOG_KPI } from "./definitions/backlog.kpi";

SchemaRegistry.register(BACKLOG_KPI);
// Done. The KPI Engine will now compute backlog for every filter combination.
// /api/kpi/summary will include "backlog" in its response automatically.
Add the source column to the DataSet schema
// Ensure "Backlog Value" is declared in your domain's DataSet column schema
const PIPELINE_SCHEMA: ColumnSchema[] = [
  // ... existing columns ...
  { name: "Backlog Value", type: "number", required: false, aliases: ["Backlog $M", "BacklogVal"] },
];

The backlog KPI is now available in every API response, the PM Report, the chat agent (via the IntentParser's KPI vocabulary), and the InsightEngine's threshold checks — with zero changes to any platform code.

03 · Add a New Business Domain

Writing a DomainConfig Config Only

A DomainConfig is a plain object that maps a business domain to the platform's abstract schema. It specifies which connectors to use, which KPIs to activate, which insight rules to apply, and which personas to expose. The platform code does not change. 

// src/domains/ibm-procurement/domain.config.ts
import { DomainConfig } from "../../platform/domain";

export const PROCUREMENT_DOMAIN: DomainConfig = {
  id: "procurement",
  displayName: "Procurement Intelligence",
  version: "1.0.0",

  // ── Data sources ──
  dataSources: [
    {
      connectorId: "rest",
      role: "purchase_orders",
      config: {
        baseUrl: "https://your-procurement-api.example.com/api/v2",
        authType: "bearer",
        tokenEnvVar: "ARIBA_API_TOKEN",
      },
      query: { endpoint: "/purchase-orders", params: { status: "open,closed" } },
      refreshIntervalMs: 3_600_000,
    },
    {
      connectorId: "db2",
      role: "supplier_master",
      config: {
        host: process.env.DB2_HOST,
        database: "PROCUREMENT",
      },
      query: { table: "SUPPLIER_MASTER" },
      refreshIntervalMs: 86_400_000,   // daily
    }
  ],

  // ── KPIs to activate ──
  kpiIds: ["po_volume", "supplier_lead_time", "cost_savings", "contract_coverage"],

  // ── Dimension mapping (source column names → platform dimension names) ──
  dimensionMap: {
    geo:          "BuyerRegion",
    market:       "BuyerCountry",
    offeringArea: "SpendCategory",
    quarter:      "FiscalQuarter",
    year:         "FiscalYear",
  },

  // ── Personas ──
  personas: [
    { id: "procurement_lead",  displayName: "Procurement Lead",  defaultFilters: { geo: "WW" } },
    { id: "cpo",               displayName: "Chief Procurement Officer", defaultFilters: { geo: "WW", quarter: "CURRENT" } },
  ],

  // ── Domain-specific insight rules (in addition to platform defaults) ──
  domainInsightRules: [
    { id: "supplier_concentration", threshold: 0.4, description: "Alert if top-3 suppliers hold >40% of spend" },
    { id: "lead_time_degradation",  threshold: 10,  description: "Flag suppliers with lead time increase >10 days QoQ" },
  ],

  // ── Knowledge base entries for chat agent ──
  knowledgeBase: [
    { topic: "spot_buy",   content: "A spot buy is a one-time purchase outside contracted agreements..." },
    { topic: "renegotiate", content: "Renegotiation is triggered when contract coverage drops below 70%..." },
  ],

  // ── Narrator persona prompt overrides ──
  narratorPersonaOverrides: {
    cpo: "You are a procurement intelligence advisor for the Chief Procurement Officer. Focus on cost savings, supplier risk, and contract coverage. Always quantify savings in $M.",
  },
};
Registration
DomainRegistry.register(PROCUREMENT_DOMAIN) — then start the server. The platform bootstraps the connectors, activates the specified KPIs, and exposes all API endpoints scoped to this domain at /api/domains/ibm_procurement/.... No other code change required.
04 · Add a New Insight Rule

Writing an InsightRuleDef

Insight rules are declarative functions that receive a KpiSnapshot and return a typed Insight | null. They are pure functions — no side effects, no external calls. The InsightEngine runs every registered rule and aggregates the results into the InsightBundle. 

// src/insights/rules/pipeline-stall.rule.ts
import { InsightRule, KpiSnapshot, Insight } from "../../platform/insights";

/**
 * PipelineStallRule — fires when active pipeline has not grown for 2+ consecutive quarters.
 * Severity scales with stagnation depth: >10% decline = HIGH, flat = MED.
 */
export const PIPELINE_STALL_RULE: InsightRule = {
  id: "pipeline_stall",
  displayName: "Pipeline Stagnation Detector",
  appliesTo: ["pipeline"],   // which KPI this rule fires on

  evaluate(snapshot: KpiSnapshot): Insight | null {
    const { current, priorQuarter, priorPriorQuarter } = snapshot.pipeline;

    // Need at least 2 prior quarters to detect stagnation
    if (!priorQuarter || !priorPriorQuarter) return null;

    const qoq1 = (current - priorQuarter) / priorQuarter;      // current vs PQ
    const qoq2 = (priorQuarter - priorPriorQuarter) / priorPriorQuarter;  // PQ vs PPQ

    // Two consecutive declines
    if (qoq1 < 0 && qoq2 < 0) {
      const severity = qoq1 < -0.10 ? "HIGH" : "MED";
      return {
        id: `pipeline_stall_${snapshot.scope.geo}_${snapshot.scope.quarter}`,
        ruleId: "pipeline_stall",
        severity,
        headline: `${snapshot.scope.geo} pipeline declining for 2 consecutive quarters (${(qoq1*100).toFixed(1)}% QoQ, ${(qoq2*100).toFixed(1)}% prior QoQ)`,
        supportingData: { current, priorQuarter, priorPriorQuarter, qoq1, qoq2 },
        confidence: snapshot.dataQuality.completeness,  // 0–1 from DataSet
      };
    }
    return null;  // rule did not fire — not an insight
  }
};
// Register in InsightEngine (or domain config)
InsightEngine.registerRule(PIPELINE_STALL_RULE);
// The rule now runs automatically on every InsightEngine.analyse() call.
Testing insight rules
Because rules are pure functions (KpiSnapshot in, Insight out), they are trivially unit-testable without mocking any infrastructure. 
const insight = PIPELINE_STALL_RULE.evaluate(MOCK_SNAPSHOT_TWO_DECLINES);
expect(insight?.severity).toBe("HIGH");
expect(insight?.headline).toContain("2 consecutive quarters");
05 · Add a New Agent

Implementing the Agent Interface

All agents implement the same typed interface. The Orchestrator discovers agents via the AgentRegistry and routes to them based on IntentResult.agentTarget. 

// src/agents/ForecastRiskAgent.ts — a new agent for forecast risk analysis
import { BaseAgent, AgentResult, RunTrace } from "../platform/agents";
import { InsightEngine } from "../platform/insights";

interface ForecastRiskContext {
  snapshot: KpiSnapshot;
  quarter: string;
  geo: string;
}

interface ForecastRiskResult {
  coverageRatio: number;
  atRiskPipeline: number;
  riskLevel: "HIGH" | "MED" | "LOW";
  topRiskyOpportunities: Opportunity[];
  recommendation: string;
}

export class ForecastRiskAgent extends BaseAgent<ForecastRiskContext, ForecastRiskResult> {
  readonly id = "forecast_risk";
  readonly version = "1.0.0";

  // BaseAgent.execute() wraps this in try/catch, emits RunTrace, measures latency
  protected async run(ctx: ForecastRiskContext): Promise<ForecastRiskResult> {
    const coverage = ctx.snapshot.call / ctx.snapshot.pipeline;
    const atRisk   = ctx.snapshot.upside + ctx.snapshot.bestCase;

    // Pure deterministic computation — no LLM call here
    const riskLevel = coverage < 0.5 ? "HIGH" : coverage < 0.7 ? "MED" : "LOW";

    const topRisky = await this.getTopRiskyOpps(ctx);

    return {
      coverageRatio: coverage,
      atRiskPipeline: atRisk,
      riskLevel,
      topRiskyOpportunities: topRisky,
      recommendation: this.buildRecommendation(riskLevel, coverage, ctx.geo),
    };
  }

  async healthCheck() {
    return { healthy: true, agentId: this.id, version: this.version };
  }
}
// Register with the Orchestrator
AgentRegistry.register(new ForecastRiskAgent());

// Add routing in IntentParser so chat can trigger it
IntentParser.addRoute({
  pattern: /forecast risk|coverage risk|at.?risk/i,
  agentTarget: "forecast_risk",
});
06 · Switch or Add a Narrator (LLM)

Implementing NarratorService

The platform ships with LlmNarrator (any LLM provider) and TemplateNarrator (deterministic fallback). To add GPT-4, Llama 3, or any other LLM: implement NarratorService, update the binding. Zero other changes. 

// src/narrators/Gpt4Narrator.ts
import { NarratorService, NarrativeResult, InsightBundle, NarratorContext } from "../platform/narrator";
import OpenAI from "openai";

export class Gpt4Narrator implements NarratorService {
  private client = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

  async narrate(bundle: InsightBundle, ctx: NarratorContext): Promise<NarrativeResult> {
    const prompt = PromptBuilder.build(bundle, ctx);

    const response = await this.client.chat.completions.create({
      model: "gpt-4o",
      temperature: 0,                          // deterministic
      response_format: { type: "json_object" }, // enforce JSON output
      messages: [
        { role: "system", content: prompt.system },
        { role: "user",   content: JSON.stringify(bundle) },
      ],
      max_tokens: 600,
    });

    const parsed = JSON.parse(response.choices[0].message.content!);

    // OutputValidator: every number in parsed must exist in bundle
    const validation = OutputValidator.validate(parsed, bundle);
    if (!validation.passed) {
      // Automatic fallback — never propagate a bad LLM response
      return TemplateNarrator.narrate(bundle, ctx);
    }

    return {
      text: parsed.body,
      structuredFallback: bundle,    // always present
      llmUsed: true,
      validationPassed: true,
      tokensUsed: response.usage?.total_tokens,
    };
  }
}
// Swap the binding in the DI container — one line change
Container.bind(NarratorService).to(Gpt4Narrator);
// LlmNarrator is replaced; TemplateNarrator fallback is unchanged.
// All agents, all API endpoints, all chat flows continue to work.
Extension Checklist

Before Opening a Pull Request

Every extension must pass this checklist before merging.

  • New connector/agent/rule implements its interface — not a concrete dependency
  • New connector passes ConnectorContractTest (12 standard assertions)
  • New insight rule is a pure function with unit tests — no infrastructure mocks needed
  • New KpiDefinition has at least one threshold rule and one insight rule
  • New domain config has a dimension map covering geo, quarter, offeringArea
  • New agent emits RunTrace on every execution (success and failure)
  • New endpoint added to the golden eval test set (evals/golden.json)
  • Eval pass rate ≥95% after changes (run npm run eval)
  • No new imports from a higher layer into the modified layer
  • PR description includes: layer modified, interface implemented, zero-touch guarantee confirmed
