00 · High-Level Plan

The 7-Point Platform Blueprint

If a world-class AI Product Leader, Data Science Engineer, and Software Architect co-designed Compass from scratch as a platform — not a single-purpose app — these are the seven structural decisions they would make before writing a single line of code.

Core Design Thesis

The non-deterministic nature of LLMs is the single biggest risk to enterprise AI trust. The platform's answer: LLMs are optional and late-stage. Every KPI calculation, every trend, every anomaly, every recommendation is computed deterministically first. The LLM is a narrator, not a calculator. The structured output is always the source of truth.

1 · Source-Agnostic Data Contract

All data enters through a single DataConnector interface. Whether the source is Excel, DB2, Oracle, Snowflake, a REST API, or a data lake, the platform always receives normalized DataSet objects. Zero upstream assumptions in any downstream layer.

2 · Schema-Driven KPI Engine

KPI definitions are declared in a JSON schema — not hard-coded in route handlers. Adding a new KPI is adding a schema entry, not writing a new API endpoint. The engine executes the schema; it does not embed it.

3 · Deterministic Intelligence Layer

Every insight (trend, anomaly, risk, recommendation) is produced by a deterministic rule engine with a typed InsightResult output. This layer has 100% unit-testable, reproducible output. LLMs only convert structured output to natural language — they never produce the numbers.

4 · Typed Agent Contract

Every agent (Data Agent, Analytics Agent, Insight Agent, GTM Agent, Memory Agent) implements a standard Agent interface: execute(context) → AgentResult. Agents are composable, replaceable, and independently testable. Watsonx Orchestrate routes between them.

5 · LLM as Isolated Narrator

The LLM receives only pre-validated, structured JSON. Its sole responsibility is to produce human-readable language from that JSON. It cannot call tools, modify data, or affect downstream state. Output is validated against a schema before it ever reaches the client.

6 · Observability as a First-Class Citizen

Every agent execution produces a RunTrace object: inputs, outputs, latency, confidence score, data lineage, and LLM token cost. This is not a bolt-on — it is part of the Agent base class. The trace is the audit log, the eval dataset, and the ROI proof.

7 · Domain-Configurable, Not Domain-Specific

The platform is deployed once. A domain config object maps any business domain (Sales KPIs, HR metrics, Supply Chain, Finance P&L) to the platform's abstract schema. Reuse is a configuration problem, not a code problem. The same platform serves sales performance today, and procurement analytics tomorrow, with zero change to platform code.

01 · Design Principles

Five Non-Negotiable Principles

Determinism First

Any output that includes a number must be reproducible. Given the same input data, the platform produces the exact same KPI values, trend deltas, and recommendations — every time. LLM text may vary; structured data never does.

Accuracy by Contract

Every DataConnector validates against a schema. Every aggregation has an associated AccuracyCheckpoint that cross-validates totals. Numbers presented in the UI are checksummed against the source layer before display.

Resiliency by Default

No single point of failure. If the LLM is unavailable, the platform falls back to template-based narratives. If a data source is stale, the UI shows a data-freshness warning with the last-known timestamp. Degraded mode is always better than failure.

Efficiency through Caching & Pre-computation

KPI summaries for standard filter combinations (WW, current quarter, all geos) are pre-computed at data load time and served from an in-memory cache. The vast majority of requests are cache hits. LLM calls are never made for repeat queries within a session TTL.

Abstraction over Implementation

Every dependency is accessed through an interface, not a concrete class. DataConnector, InsightEngine, NarratorService, MemoryStore — all are interfaces first. This is what makes the platform reusable: swap the implementation, keep the contract.

02 · Architecture

7-Layer Platform Architecture

The platform is organized into seven vertical layers. Data flows strictly downward. No layer may call upward. Every cross-layer call is mediated by a typed interface. See architecture.html for the full visual diagram.

Key Architectural Constraint

Data flows strictly downward: Source → Connector → KPI Engine → Intelligence → Narrator → Agent Bus → Presentation. No layer may import from a layer above it. All cross-layer communication is through typed interfaces and immutable value objects. This constraint is what makes individual layers independently testable, deployable, and replaceable.

03 · Object-Oriented Abstractions

Core Interfaces & Value Objects

Every reusable platform component is defined as an interface first. Concrete implementations are injected — they are never instantiated inline. This is the structural reason the platform can serve multiple domains and multiple data sources.

DataConnector — The Source Abstraction

/**
 * DataConnector — implemented once per source type.
 * The platform never sees raw source data; it only sees DataSet objects.
 */
interface DataConnector {
  readonly id: string;                   // "excel" | "db2" | "snowflake" | "rest" | ...
  readonly displayName: string;

  /** Test connection; throw ConnectorError if unreachable */
  connect(): Promise<void>;

  /**
   * Ingest source and emit a normalized DataSet.
   * @param query  — optional source-specific filter (SQL WHERE, API params, sheet name, etc.)
   * @param schema — the ColumnSchema the caller expects; connector must validate and coerce
   */
  fetch(query: ConnectorQuery, schema: ColumnSchema[]): Promise<DataSet>;

  /** Return metadata: last-modified timestamp, row count estimate, connection health */
  status(): Promise<ConnectorStatus>;

  disconnect(): Promise<void>;
}

// Concrete implementations:
class ExcelConnector     implements DataConnector { ... }
class Db2Connector       implements DataConnector { ... }
class OracleConnector    implements DataConnector { ... }
class SnowflakeConnector implements DataConnector { ... }
class RestApiConnector   implements DataConnector { ... }
class DataLakeConnector    implements DataConnector { ... }  // any SQL data lake
class SalesforceConnector  implements DataConnector { ... }  // ISC / CRM

DataSet — The Normalized Value Object

/**
 * DataSet is the universal currency between Layer 0/1 and Layer 2.
 * Once data is a DataSet, the KPI Engine does not care about its origin.
 */
interface DataSet {
  readonly id: string;                   // uuid — for cache keying and lineage tracking
  readonly sourceId: string;             // which connector produced it
  readonly fetchedAt: Date;              // used for data-freshness warnings
  readonly schema: ColumnSchema[];       // typed column definitions
  readonly rows: ReadonlyArray<Row>;     // normalized, coerced, validated rows
  readonly rowCount: number;
  readonly checksum: string;             // SHA-256 of serialized rows — for accuracy verification
  readonly warnings: DataWarning[];      // e.g. missing columns, partial data, coercion counts
}

// Column types — all values coerced at ingestion, never at query time
type ColumnType = "string" | "number" | "date" | "boolean" | "enum";
interface ColumnSchema {
  name: string;
  type: ColumnType;
  required: boolean;
  enumValues?: string[];       // for "enum" type — validated at ingest
  aliases?: string[];          // alternative source column names (Excel header variants, etc.)
}

KpiDefinition — The Schema-Driven KPI

/**
 * KPIs are declared, not coded. The KPI Engine reads this schema
 * and executes it — it does not contain KPI-specific logic.
 * Adding a new KPI = adding a KpiDefinition entry.
 */
interface KpiDefinition {
  id: string;                        // "revenue" | "signings" | "pipeline" | "win_rate"
  displayName: string;
  unit: "currency_usd" | "count" | "percentage" | "ratio";
  aggregation: AggregationFn;        // sum | count | ratio | weighted_average
  sourceColumn: string;              // maps to DataSet column name
  filterDimensions: string[];        // which dimensions can slice this KPI
  comparisons: ComparisonDef[];      // QoQ, YoY, MoM — computed automatically
  thresholds: ThresholdRule[];       // defines what "at risk" means for this KPI
  insightRules: InsightRuleDef[];    // deterministic insight rules for this KPI
}

// Example declaration (no code change needed to add Backlog, NPS, etc.):
const REVENUE_KPI: KpiDefinition = {
  id: "revenue",
  displayName: "Revenue Actuals",
  unit: "currency_usd",
  aggregation: "sum",
  sourceColumn: "Rev Act @ PC",
  filterDimensions: ["geo", "market", "country", "quarter", "year", "offeringArea"],
  comparisons: [
    { type: "QoQ", label: "vs Prior Quarter" },
    { type: "YoY", label: "vs Prior Year Quarter" }
  ],
  thresholds: [
    { condition: "yoy_delta < -10", severity: "HIGH",  label: "Revenue declining YoY" },
    { condition: "qoq_delta < -5",  severity: "MED",   label: "Revenue declining QoQ" },
  ],
  insightRules: [
    { id: "rev_anomaly_large_deal", description: "Detect prior-period one-off deals inflating YoY base" },
    { id: "rev_geo_concentration", description: "Flag if top-1 geo accounts for >50% of total revenue" },
  ]
};

Agent — The Agentic Abstraction

/**
 * Every agent in the platform — Data, Analytics, Insight, GTM, Memory —
 * implements this single interface. The Orchestrator calls execute();
 * it never knows which agent it is calling.
 */
interface Agent<TContext, TResult> {
  readonly id: string;
  readonly version: string;

  /** Primary execution — always returns a typed result, never throws */
  execute(context: TContext): Promise<AgentResult<TResult>>;

  /** Health check — called by Orchestrator before routing */
  healthCheck(): Promise<AgentHealth>;
}

interface AgentResult<T> {
  success: boolean;
  data: T | null;
  error: AgentError | null;
  trace: RunTrace;                  // ALWAYS emitted — even on failure
}

interface RunTrace {
  agentId: string;
  traceId: string;                  // uuid — correlates across multi-agent chains
  startedAt: Date;
  durationMs: number;
  inputSummary: Record<string, unknown>;
  outputSummary: Record<string, unknown>;
  confidence: number;               // 0–1 — deterministic score, not LLM probability
  confidenceFactors: string[];      // human-readable reasons for the confidence score
  dataLineage: string[];            // DataSet ids consumed
  llmUsed: boolean;
  llmTokensUsed?: number;
  llmCostUsd?: number;
  fallbackActivated: boolean;
  warnings: string[];
}

InsightEngine — Deterministic Intelligence

/**
 * InsightEngine is deterministic. Given the same KpiSnapshot, it always
 * produces the same InsightBundle. No randomness, no LLM calls.
 * This is the accuracy and reproducibility guarantee.
 */
interface InsightEngine {
  analyse(snapshot: KpiSnapshot, rules: InsightRuleDef[]): InsightBundle;
}

interface InsightBundle {
  readonly snapshotId: string;      // links back to the KpiSnapshot that produced it
  readonly computedAt: Date;
  readonly highlights: Insight[];   // what's going well
  readonly risks: Insight[];        // what needs attention
  readonly anomalies: Insight[];    // statistical outliers
  readonly recommendations: Recommendation[];  // ranked by expected $M impact
}

interface Insight {
  id: string;
  ruleId: string;                   // which InsightRuleDef fired
  severity: "HIGH" | "MED" | "LOW" | "INFO";
  headline: string;                 // concise, numbers-included string (deterministic)
  supportingData: Record<string, unknown>;  // the raw numbers behind the headline
  confidence: number;               // 0–1 — based on data completeness, sample size
}

interface Recommendation {
  id: string;
  category: "pipeline" | "win_rate" | "revenue" | "forecast" | "gtm";
  geo: string;
  action: string;                   // deterministic action string
  expectedImpactM: number | null;   // estimated $M impact (rule-based, not LLM)
  confidence: number;
  priority: "HIGH" | "MED" | "LOW";
  deadline: string;
}

NarratorService — The LLM Isolation Boundary

/**
 * The LLM is behind this interface. If the LLM is down, TemplateNarrator
 * activates automatically. The caller never knows the difference — it
 * always receives a NarrativeResult with the same shape.
 */
interface NarratorService {
  narrate(bundle: InsightBundle, context: NarratorContext): Promise<NarrativeResult>;
}

interface NarratorContext {
  persona: "executive" | "pm" | "analyst";
  language: string;                 // "en" | "de" | "ja" | ...
  lengthHint: "brief" | "standard" | "detailed";
  domainConfig: DomainConfig;       // provides domain-specific terminology
}

interface NarrativeResult {
  text: string;                     // the human-readable narrative
  structuredFallback: InsightBundle; // ALWAYS present — the source of truth
  llmUsed: boolean;                 // false if TemplateNarrator activated
  validationPassed: boolean;        // output was schema-validated
  tokensUsed?: number;
}

// LlmNarrator: calls any LLM (OpenAI, Anthropic, Granite, Llama) with structured prompt
// TemplateNarrator: produces deterministic text from InsightBundle templates (zero LLM)
// OutputValidator: validates that LLM output contains all required numbers from the bundle
04 · Solving Non-Determinism

How the Platform Eliminates AI Non-Determinism

Non-determinism in enterprise AI is not just a technical problem — it's a trust problem. If two executives ask the same question and get different numbers, the product is dead. The platform's answer is a four-layer determinism guarantee.

The Problem with Naive LLM Integration
  • LLM asked to analyse data → hallucinates numbers
  • LLM called twice → different answers
  • LLM prompt changes → all historical outputs become inconsistent
  • LLM temperature > 0 → non-reproducible text
  • LLM downtime → complete product failure
The Platform's Solution
  • Compute first, narrate second — all numbers from KPI Engine, never LLM
  • Structured prompt contract — LLM receives JSON, must return JSON
  • OutputValidator — every LLM response is schema-validated; if it fails, TemplateNarrator activates
  • Temperature = 0 — all LLM calls use deterministic sampling
  • Fallback chain — LLM → TemplateNarrator → raw structured data

The Structured Prompt Contract

The LLM is never given raw data and asked "what do you think?". It is given a validated InsightBundle JSON and a strict output schema, and asked to produce language only. 

// LlmNarrator sends this — never raw data, never open-ended questions
const prompt = {
  system: `You are an executive business narrator. You receive a structured InsightBundle JSON.
           Your ONLY job is to convert the pre-computed data into clear, concise executive language.
           You must NOT compute, invent, or modify any numbers. Every statistic in your output
           must appear verbatim in the InsightBundle you received.
           Respond in this JSON schema: { "headline": string, "body": string, "bullets": string[] }`,

  user: JSON.stringify(insightBundle),   // validated, typed, checksummed

  config: {
    temperature: 0,                      // fully deterministic sampling
    maxTokens: 500,
    responseFormat: "json_object",       // enforce JSON output mode
  }
};

// After receiving LLM response:
const validated = OutputValidator.validate(response, insightBundle);
// OutputValidator checks: every number in the narrative appears in the InsightBundle.
// Any number in the narrative NOT in the InsightBundle → validation failure → fallback.
if (!validated.passed) {
  return TemplateNarrator.narrate(insightBundle, context);
}

LLM downtime Product completely unavailable Automatic TemplateNarrator fallback; structured data always returned Layer 4 Narrator

06 · Agent Design

Five Specialized Agents + Orchestrator

Each agent has a single, narrow responsibility. The Orchestrator routes between agents based on the IntentResult from the IntentParser. No agent calls another agent directly — all inter-agent communication is through the Orchestrator.

DataAgent
Responsibility: Source → DataSet

Manages all DataConnector instances. Handles scheduling, refresh triggers, staleness detection, and the raw DataSet cache. Emits DataReadyEvent when a new DataSet is available — downstream agents subscribe.

AnalyticsAgent
Responsibility: DataSet → KpiSnapshot

Owns the KPI Engine. Receives a DataSet and a filter Context, executes the SchemaRegistry, and returns a validated, checksummed KpiSnapshot. All aggregation, delta computation, and AccuracyCheckpoints happen here.

InsightAgent
Responsibility: KpiSnapshot → InsightBundle

Runs the deterministic InsightEngine: TrendAnalyser, AnomalyDetector, RiskScorer. Produces a fully typed, confidence-scored InsightBundle. No LLM. Outputs are reproducible given the same KpiSnapshot.

GtmAgent
Responsibility: InsightBundle → Recommendations

Converts insights into ranked GTM recommendations with expected $M impact. Draws on the MemoryStore to surface historically successful actions for similar KPI patterns. Rules-based first; LLM refines language only.

ReviewAgent
Responsibility: InsightBundle + History → Narrative

Coordinates the NarratorService to produce the executive narrative, PM report, and MBR/QBR decks. Manages the fallback chain (LLM → TemplateNarrator). Owns output validation and the NarrativeResult response.

MemoryAgent
Responsibility: insight → action → outcome schema

Stores every InsightBundle, the action taken in response, and the observed outcome (from next-period KPIs). This is the compounding moat: each cycle, recommendations improve because they draw on an ever-growing library of what worked.

Intent Parser — Deterministic Chat Routing

/**
 * The IntentParser is purely deterministic — regex + rule-based NLU.
 * It routes every chat message to the correct agent with a typed context.
 * The LLM is never in the chat routing path.
 * This eliminates prompt injection and guarantees consistent routing.
 */
class IntentParser {
  parse(message: string, session: SessionContext): IntentResult {
    // 1. Extract filter dimensions (geo, quarter, offeringArea, ut30)
    // 2. Classify intent: pipeline | revenue | signings | win_rate | forecast |
    //                      gtm_action | narrative | domain_knowledge
    // 3. Return typed IntentResult — Orchestrator uses this to route to the correct agent
    return {
      intent: "pipeline_summary",
      confidence: 0.97,
      filters: { geo: "EMEA", quarter: "2Q26" },
      agentTarget: "AnalyticsAgent",
      fallbackChain: ["InsightAgent", "ReviewAgent"],
    };
  }
}
