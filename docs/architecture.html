Determinism Guarantee — Layer by Layer

Layer 1 SHA-256 checksum on every DataSet. Identical source data always produces identical DataSet.
Layer 2 Pure functions only in AggregationEngine. AccuracyCheckpoint cross-validates UI totals vs API sums. Same inputs → identical KpiSnapshot every time.
Layer 3 Rule-based engine. No probabilistic models. Every InsightBundle is reproducible given the same KpiSnapshot.
Layer 4 LLM called with temperature=0 + JSON mode. OutputValidator: every number in LLM output must appear in InsightBundle. Failure → TemplateNarrator activates automatically.
Layer 5 IntentParser is pure regex/rule NLU — no LLM in routing path. Identical message + session context → identical intent → identical agent route.

Agent Routing Flow — Chat Request
// User sends: "How is EMEA signings this quarter?"

Chat UI             → POST /api/chat { message, sessionId }
                      ↓
IntentParser         → IntentResult { intent:"signings", geo:"EMEA", quarter:"3Q26" }
                      ↓ (deterministic — no LLM)
Orchestrator         → routes to AnalyticsAgent
                      ↓
AnalyticsAgent      → KpiEngine.compute(EMEA, 3Q26, "signings")
                      ↓ KpiSnapshot (deterministic)
InsightAgent        → InsightEngine.analyse(snapshot) → InsightBundle
                      ↓ (deterministic)
ReviewAgent         → NarratorService.narrate(bundle, {persona:"pm"})
                      ↓ LlmNarrator (temp=0) OR TemplateNarrator
Chat UI             ← NarrativeResult { text, structuredFallback, trace }
Cross-Cutting Concerns
Observability — RunTrace emitted by every agent. Feeds agent-runs.jsonl, admin dashboard, and eval harness.
Caching — KpiCache stores pre-computed snapshots for standard filter combinations. P95 cache-hit response: <200ms.
Data Freshness — Every DataSet carries fetchedAt. UI shows staleness warning if data age exceeds domain threshold.
Fallback Chain — LLM → TemplateNarrator → raw InsightBundle JSON. Product never fully fails.
Validation — AccuracyCheckpoint at Layer 2. OutputValidator at Layer 4. Schema validation at Layer 1.
Deployment Topology
Container Platform — any serverless or container hosting (Fly.io, Railway, Render, AWS ECS). Recommended for API + SPA.
Container Registry — Docker image at ghcr.io/puru-nayak-labs/compass-app
Data Layer — ingestion + KPI memory store (Phase 2+). Any SQL-compatible data warehouse.
LLM (any provider) — narrator layer. Called only for narrative generation, never for computation.
Agent Orchestrator — agent routing bus (Phase 3). Manages multi-agent chains and skill registration.
Chat Interface — chat frontend. Session-aware. Routes to Orchestrator or direct REST API.
Memory Agent Schema

The long-term moat. Compounds with every business cycle.

MemoryRecord {
  id: uuid
  period: "3Q26"
  geo: "EMEA"
  insight: InsightBundle
  action: Recommendation
  outcome: KpiDelta (next period)
  actionTaken: boolean
  outcomeValue: number
  recordedAt: Date
}
Eval Harness
Golden test set — 30–50 question/answer pairs covering every chat intent
Eval runner — POST each golden case to /api/chat; assert pass/fail; target ≥95%
Pre-commit gate — CI fails if eval pass rate drops below threshold
Thumbs-down capture — auto-appends full RunTrace to failures.jsonl for offline review
KPI Schema Registry (Declarative)

Adding a KPI = adding a schema entry. Zero code change.

KpiDefinition {

  aggregation: sum|count|ratio
  sourceColumn: string
  filterDimensions: string[]

}
